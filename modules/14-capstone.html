Cómo trabajar con un capstone How to work on a capstone

Cada capstone tiene cuatro componentes:

  1. Specification — el qué y el porqué. Problema, criterios de aceptación, deliverables.
  2. Starter repo — esqueleto reproducible con tests, CI, dependencias fijadas. Clonas y empiezas.
  3. Reference solution — implementación de referencia (no la única correcta) para comparar tu enfoque.
  4. Rubric — autoevaluación basada en criterios objetivos: pasa tests, métricas de calidad, métricas de coste, documentación.

Tiempo recomendado: 8-15 horas focused trabajando en sesiones de 2-3 horas. NO intentes hacerlo de una sentada — el aprendizaje viene del ciclo iterativo de probar, fallar, ajustar.

Lo más valioso: el writeup final. Tras terminar, escribe un README de 500-800 palabras que explique tu enfoque, qué probaste que no funcionó, qué métricas obtuviste y qué harías distinto la próxima vez. Eso es lo que un empleador serio quiere ver.

Every capstone has four components:

  1. Specification — the what and why. Problem, acceptance criteria, deliverables.
  2. Starter repo — reproducible skeleton with tests, CI, pinned dependencies. Clone and go.
  3. Reference solution — reference implementation (not the only correct one) to compare your approach.
  4. Rubric — self-assessment on objective criteria: tests pass, quality metrics, cost metrics, documentation.

Recommended time: 8-15 focused hours in 2-3 hour sessions. DON'T try to ship in a single sitting — the learning happens in the test-fail-tune cycle.

The most valuable artefact: the final writeup. After shipping, write a 500-800 word README explaining your approach, what you tried that didn't work, what metrics you got and what you'd do differently next time. That's what a serious employer wants to read.

Proyecto A — Code Review Bot end-to-end Project A — End-to-end Code Review Bot

📋 Specification

Problema: revisar Pull Requests manualmente consume 30-45 min por PR de un dev senior. Construir un bot que revise PRs automáticamente al abrirse, deje comentarios línea-por-línea con severidad, y sirva de "primer pase" antes del review humano.

Criterios de aceptación:

  • Trigger automático en GitHub Actions cuando se abre/actualiza un PR.
  • Detecta al menos 4 categorías de problemas: SECURITY (OWASP Top 10), CORRECTNESS (lógica), PERFORMANCE (N+1, leaks), MAINTAINABILITY (legibilidad).
  • Output structured (JSON Schema) con: file, line, severity (critical/high/medium/low), issue, suggested_fix, confidence (0-100%).
  • Comenta solo findings con confidence ≥ 70 — el resto van a una sección "low-confidence" expandible.
  • Sobre el golden dataset de 50 PRs reales (proporcionado), recall ≥ 80% en críticos y precision ≥ 85%.
  • Coste medio ≤ $0.30 por PR (medido).
  • P95 latencia ≤ 90 segundos desde PR abierto hasta primer comentario publicado.
🛠️ Stack sugerido
  • Lenguaje: Python 3.11+.
  • LLM API: Anthropic Claude Opus 4.7 (mejor en code review). Fallback secundario: GPT-5 o DeepSeek V4 Pro.
  • Integración GitHub: octokit via PyGithub o llamadas REST directas. Webhook desde GitHub Actions.
  • Caching: prompt caching de Anthropic con system prompt + ejemplos few-shot cacheados (5 min TTL es suficiente para una serie de PRs).
  • Tests: pytest + dataset de 50 PRs con golden answers preparados.
  • Observabilidad: log estructurado JSON + métricas a Prometheus o Helicone.
📦 Deliverables
  1. code-review-bot/ repo con código + GitHub Action .github/workflows/review.yml.
  2. System prompt versionado con XML structure y 5 few-shot examples.
  3. JSON Schema de findings + validación strict.
  4. Eval suite: tests/eval_golden.py que corre los 50 PRs y reporta recall/precision/coste.
  5. README con: setup, decisiones de diseño, métricas obtenidas, limitaciones conocidas.
  6. Demo en GIF o screencast de un PR real revisado por el bot.
⚠️ Trampas conocidas (lecciones de los que ya lo hicieron)
  • Diff completo en el prompt explota tokens. Filtra hunks irrelevantes (whitespace-only, generated code) antes de enviar.
  • Modelos alucinan números de línea. No confíes ciegamente — valida que el line number exista en el diff antes de publicar.
  • Falsos positivos en seguridad ahogan al dev. Mejor 5 critical-findings reales que 30 "esto podría ser SQL injection en teoría".
  • Coste se dispara con prompts gigantes. Si el PR es ≥ 1000 líneas, divide por archivo y revisa por chunks.
  • Sin prompt caching, $1+ por PR. Con caching del system prompt, baja a $0.10-0.30. Marca cache_control en system + ejemplos.

Proyecto B — RAG Knowledge Assistant Project B — RAG Knowledge Assistant

📋 Specification

Problema: tu organización tiene 500-5000 documentos (Confluence, Notion, Drive, PDFs, manuales internos). Un nuevo empleado tarda semanas en orientarse. Construye un asistente que responda preguntas sobre esa base de conocimiento, citando fuentes verificables.

Criterios de aceptación:

  • Ingesta automatizada de un corpus inicial de ≥ 200 documentos (≥ 2 MB de texto plano tras parsing).
  • Pipeline RAG completo: chunking + embeddings + vector DB + retrieval + augmented generation + citation.
  • Cada respuesta incluye al menos 2 citaciones [doc:X, sección:Y] con enlaces clicables al documento original.
  • Si la respuesta no está en el corpus, el bot dice "No encontrado en las fuentes" — sin alucinar.
  • Sobre un eval set de 30 preguntas con golden answers, accuracy ≥ 80% y citation accuracy ≥ 90% (la cita realmente respalda la afirmación).
  • Latencia p95 ≤ 5 segundos por respuesta.
  • Coste medio ≤ $0.05 por respuesta.
🛠️ Stack sugerido
  • Lenguaje: Python 3.11+ (frontend opcional Next.js / Streamlit).
  • Embeddings: OpenAI text-embedding-3-small ($0.02/1M tok) o BGE-M3 self-hosted (gratis).
  • Vector DB: Qdrant self-hosted o Chroma local para empezar; Pinecone Serverless si quieres cero ops.
  • Generation: Claude Sonnet 4.6 (balance calidad/coste/contexto) o DeepSeek V4 Pro.
  • Chunking: RecursiveCharacterTextSplitter (LangChain) con chunk_size=512, overlap=50. O semantic chunking para corpus técnico.
  • Frameworks: LangChain o LlamaIndex para el pipeline; Instructor para outputs estructurados.
  • Eval: RAGAS (faithfulness, answer relevancy, context precision/recall).
📦 Deliverables
  1. rag-assistant/ repo con: ingesta, indexado, query, frontend.
  2. Script ingest.py que procesa una carpeta y construye el índice.
  3. API POST /query que devuelve {answer, citations[], confidence}.
  4. Frontend mínimo con caja de búsqueda y respuesta con citas clicables.
  5. Eval suite con 30 preguntas + RAGAS scores reportados.
  6. README con tabla de decisiones (qué embedding probaste, qué chunk size, qué k recuperaste, por qué).
⚠️ Trampas conocidas
  • Chunking malo arruina todo el resto. Si tus chunks cortan a mitad de oraciones o pierden contexto de cabeceras, el retrieval no compensa. Empieza con RecursiveCharacterSplitter, mide, itera.
  • Top-K demasiado bajo o alto. K=3 pierde contexto; K=20 mete ruido y dispara coste. K=5-7 es el sweet spot habitual.
  • Embeddings sin re-ranker. Para precisión seria, añade un re-ranker (Cohere Rerank o BGE-reranker) sobre los top-20 → top-5 finales. Mejora 10-20pp en context precision.
  • Citations alucinadas. Si el modelo genera [doc:42] pero ese doc no existe, el sistema rompe confianza. Valida programáticamente que cada citation referencia un chunk real.
  • Sin "No encontrado" honesto. El modelo por defecto rellena con su training data en vez de decir "no sé". Hay que prompt-engineering muy explícito para forzarlo.

Proyecto C — Local Self-Hosted Assistant Project C — Local Self-Hosted Assistant

📋 Specification

Problema: una empresa procesa datos sensibles que NO pueden salir de su red (sanidad, legal, gobierno). Construir un asistente equivalente al Proyecto B pero 100% on-premise, sin que ningún byte cruce a la nube.

Criterios de aceptación:

  • Modelo LLM corriendo localmente: Qwen3-32B Q5_K_M vía Ollama, o Llama 4 70B Q4 si tienes ≥48 GB VRAM.
  • Embeddings locales: BGE-M3 o Nomic Embed Text v2 corriendo en CPU/GPU.
  • Vector DB local: ChromaDB embebido o Qdrant local.
  • Frontend Next.js mínimo con autenticación.
  • Cero llamadas a APIs externas — verificable con monitor de red.
  • Métricas: tokens/segundo del modelo, latencia p95 ≤ 8s en hardware target documentado, calidad ≥ 75% del baseline cloud sobre el mismo eval set.
  • Documento de cumplimiento GDPR Art. 32 (medidas técnicas) preparado para auditor.
🛠️ Hardware target sugerido
  • Mínimo: RTX 4090 (24 GB VRAM) o Mac M3 Pro 36 GB unified memory → Qwen3-32B Q4_K_M, ~25 tok/s.
  • Recomendado: RTX 5090 (32 GB) o dual 4090 (48 GB) → Qwen3-32B Q5_K_M o Llama 4 70B Q4, ~20-30 tok/s.
  • Servidor: 4× A100 80GB → Llama 4 70B FP16 con vLLM, 100+ tok/s en producción.
⚠️ Trampas conocidas
  • Calidad cae con cuantizaciones agresivas. Q2/Q3 ahorran VRAM pero pierden 10-20pp de calidad. Q4_K_M es el suelo razonable; Q5_K_M es el sweet spot.
  • Latencia en CPU es brutal. Si solo tienes CPU + RAM, prepárate para 2-5 tok/s. Para uso interactivo, GPU es prácticamente obligatoria.
  • Tokenizer mismatches. Si entrenas embeddings con un tokenizer y el LLM usa otro, los chunks no coinciden semánticamente. Documenta y testea.
  • Setup de Ollama en producción. Ollama es excelente para dev pero limitado en concurrencia. Para producción seria, vLLM o SGLang son mejores.

Rúbrica completa y reference solution: próximamente Q3 2026. Si quieres priorizarlo, abre issue en el repo.

Proyecto D — Multi-Agent Research Team Project D — Multi-Agent Research Team

📋 Specification

Problema: investigar un tema técnico (p.ej. "comparativa de vector DBs en 2026") manualmente lleva 4-8 horas. Construir un equipo de agentes que haga la investigación y escriba un informe markdown publicable.

Equipo de agentes:

  • Researcher — busca en web (Tavily, Perplexity API, o Brave Search), recolecta fuentes, las anota.
  • Analyst — sintetiza las fuentes, encuentra contradicciones, extrae datos numéricos comparables.
  • Critic — red-teamea el draft, busca afirmaciones sin respaldo, sesgos, omisiones.
  • Writer — redacta el informe final en markdown publicable.
  • Supervisor — coordina, decide cuándo iterar y cuándo cerrar.

Criterios de aceptación:

  • Input: una pregunta de investigación en lenguaje natural.
  • Output: informe markdown 1500-3000 palabras con: resumen ejecutivo, secciones, tabla comparativa, citas con URLs, sección de "limitaciones de esta investigación".
  • Pipeline supervisor-worker con LangGraph o CrewAI.
  • Sobre 5 preguntas de prueba con golden reports preparados, coverage ≥ 75% (qué fuentes/datos clave aparecen) y factual accuracy ≥ 90% (afirmaciones verificables contra las URLs citadas).
  • Coste total ≤ $5 por informe.
  • Tiempo total ≤ 30 minutos por informe.
⚠️ Trampas conocidas
  • Loops infinitos entre agentes. Critic dice "falta X", Writer añade X, Critic encuentra otro problema, etc. Cap duro en iteraciones (máx 3 rondas).
  • Coste se dispara silenciosamente. 5 agentes × varios calls = decenas de llamadas. Si cada uno usa Claude Opus, el bill mensual es alarmante. Mezcla tiers: Researcher con Haiku, Writer con Opus.
  • Hallucinated URLs. El Researcher puede inventar fuentes. Verifica que cada URL devuelva 200 antes de citarla.
  • Critic demasiado complaciente o demasiado destructivo. Calibrar el system prompt del Critic es lo más difícil del proyecto.

Rúbrica completa y reference solution: próximamente Q3 2026.
