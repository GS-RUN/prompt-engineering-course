4.4-bis Authority-in-files: una postura alternativa 4.4-bis Authority-in-files: an alternative posture

🪶 Una corriente alternativa

Los patrones anteriores ponen las reglas del agente dentro del prompt — system message, skills, descripciones de herramientas, CLAUDE.md/AGENTS.md inyectados en contexto. Existe una corriente que sostiene que ése no es el sitio adecuado para la autoridad.

DMF (Drift Management Framework) saca la autoridad fuera del prompt y la coloca en archivos versionados que el agente consulta. Cuatro invariantes gobiernan cada sesión:

  1. Authority-in-files: la autoridad existe sólo en archivos. La conversación no es vinculante.
  2. Halt on conflict: cuando dos artefactos autoritativos se contradicen, paras y resuelves antes de seguir.
  3. Halt on missing authority: cuando falta un artefacto requerido, paras y lo creas o recuperas antes de seguir.
  4. Label inference: toda inferencia se etiqueta. Una inferencia sin marcar nunca es autoridad.

La apuesta: para agentes de larga vida — un agente de coding viviendo en un repo durante semanas — el drift es peor enemigo que la optimización por llamada, y los archivos son lo único que sobrevive entre sesiones.

💡 ¿Cuándo importa? A grandes rasgos: si tu unidad de trabajo es una llamada, andamia el prompt. Si es un proyecto que el agente habita, andamia el proyecto. La mayoría de despliegues acaban haciendo ambas cosas: governance en archivos por arriba, scaffolding en prompt por abajo.

📎 Repo: m-public/dmf · Spec: template/AGENTS.md

🪶 An alternative school

The patterns above put the agent's rules inside the prompt — system message, skills, tool descriptions, CLAUDE.md/AGENTS.md injected into context. There's a school of thought that says that's the wrong place for authority.

DMF (Drift Management Framework) moves authority out of the prompt into versioned project files the agent reads. Four invariants govern every session:

  1. Authority-in-files: authority exists only in files. Conversation is not binding.
  2. Halt on conflict: when authoritative artifacts disagree, stop and resolve before proceeding.
  3. Halt on missing authority: when a required artifact is absent, stop and create or recover before proceeding.
  4. Label inference: every inference must be labeled. An unlabeled inference is never authority.

The bet: for long-lived agents — a coding agent living in a repo for weeks — drift is a bigger enemy than per-call optimization, and files are the only thing that survives across sessions.

💡 When does it matter? Roughly: if your unit of work is one call, scaffold the prompt. If it's a project the agent inhabits, scaffold the project. Most production setups end up doing both: file-based governance on top, in-prompt scaffolding underneath.

4.5 Roles en Proyectos con Agentes

El Flujo de Trabajo con Agentes

  1. Setup: Creas AGENTS.md con reglas + skills/ para tareas repetitivas
  2. Task: Describes la tarea en lenguaje natural (una o dos frases)
  3. Explore: El agente lee el codebase para entender contexto
  4. Plan: Propone una estrategia (revisas y ajustas)
  5. Execute: Implementa cambios, usando sub-agentes si paraleliza
  6. Verify: Ejecuta tests, linters, corrige errores
  7. Commit: Commit + mensaje descriptivo (si se lo pides)

The Agent Workflow

  1. Setup: Create AGENTS.md with rules + skills/ for repetitive tasks
  2. Task: Describe the task in natural language (one or two sentences)
  3. Explore: The agent reads the codebase to understand context
  4. Plan: Proposes a strategy (you review and adjust)
  5. Execute: Implements changes, using sub-agents if it parallelizes
  6. Verify: Runs tests, linters, fixes errors
  7. Commit: Commit + descriptive message (if you ask for it)

18.3 Agent Swarms y Sistemas Emergentes

🌐 ¿Qué es un Agent Swarm?

Múltiples agentes simples que colaboran sin un coordinador central, siguiendo reglas locales. El comportamiento "inteligente" emerge de la interacción. Inspirado en colonias de hormigas, bancos de peces, enjambres de abejas.

Casos de Uso de Swarms

1. EXPLORACIÓN DE CÓDIGO MASIVA
   50 agentes Explore analizan 50 módulos en paralelo
   → Reporte consolidado automáticamente

2. GENERACIÓN DE DATOS SINTÉTICOS
   100 agentes generan 100 variaciones de un mismo escenario
   → Dataset de entrenamiento diverso y balanceado

3. TESTING DISTRIBUIDO
   N agentes prueban N combinaciones de inputs
   → Cobertura de edge cases imposible manualmente

4. OPTIMIZACIÓN DE PROMPTS
   20 agentes generan 20 versiones de un prompt
   → Evaluar cuál funciona mejor → iterar con las mejores
⚠️ No todo necesita multi-agente. Un solo agente bien configurado (Claude Code, OpenCode) maneja el 90% de los casos. Multi-agente añade complejidad: latencia, coste, coordinación, debugging. Úsalo solo cuando el paralelismo o la especialización justifiquen el overhead.
🌐 What is an Agent Swarm?

Multiple simple agents that collaborate without a central coordinator, following local rules. "Intelligent" behavior emerges from interaction. Inspired by ant colonies, fish schools, bee swarms.

Swarm Use Cases

1. MASSIVE CODE EXPLORATION
   50 Explore agents analyze 50 modules in parallel
   → Auto-consolidated report

2. SYNTHETIC DATA GENERATION
   100 agents generate 100 variations of the same scenario
   → Diverse, balanced training dataset

3. DISTRIBUTED TESTING
   N agents test N input combinations
   → Edge-case coverage impossible to do manually

4. PROMPT OPTIMIZATION
   20 agents generate 20 versions of a prompt
   → Evaluate which works best → iterate with the winners
⚠️ Not everything needs multi-agent. A single well-configured agent (Claude Code, OpenCode) handles 90% of cases. Multi-agent adds complexity: latency, cost, coordination, debugging. Use it only when parallelism or specialization justify the overhead.
