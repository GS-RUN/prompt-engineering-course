9.1 Structured Outputs (JSON Schema)

Garantizar formato exacto en las respuestas — crítica para producción

Claude — Tool Use + JSON Schema

// Define un "tool" con input_schema
{
  "name": "extract_entities",
  "input_schema": {
    "type": "object",
    "properties": {
      "name": {"type": "string"},
      "amount": {"type": "number"},
      "date": {"type": "string", "format": "date"}
    },
    "required": ["name", "amount"]
  }
}
// Claude rellena la estructura exacta

DeepSeek / OpenAI — JSON Mode

response_format={"type": "json_object"}
// O usar function calling:
tools=[{
  "type": "function",
  "function": {
    "name": "extract",
    "parameters": {...}
  }
}]
📐 ¿Por qué Structured Outputs?
  • Sin parsing frágil: adiós a las regex para extraer JSON de texto libre.
  • 100% de conformidad: el modelo garantiza cumplir el schema (con reintento en errores).
  • Tipado fuerte: strings, números, booleans, enums, arrays, objetos anidados.
  • Validación automática: la API rechaza respuestas que no cumplan el schema.
💡 Cuándo usar: extracción de datos, formularios, APIs, clasificación con etiquetas fijas. NO para texto creativo o respuestas abiertas.

9.2 Prompt Caching

Reduce costes hasta 90% reutilizando contexto entre llamadas

💰 Cómo Funciona
  1. Marca contenido cacheable: El system prompt, documentos largos, ejemplos
  2. Primera llamada: Se escribe en caché (coste: ~1.25× normal)
  3. Siguientes llamadas: Se lee de caché (coste: ~0.10× normal)
  4. TTL: 5-15 minutos típicamente (varía por proveedor)

9.3 Evals & Testing de Prompts

Mide objetivamente la calidad de tus prompts — no confíes en tu intuición

🧪 Framework de Evaluación
# 1. Define casos de prueba (mínimo 20-50)
test_cases = [
  {"input": "...", "expected_output": "...", "criteria": "exact_match"},
  {"input": "...", "expected_output": "...", "criteria": "contains"},
  {"input": "...", "expected_output": "...", "criteria": "semantic_similarity"},
]

# 2. Ejecuta el prompt contra todos los casos
results = [run_prompt(tc["input"]) for tc in test_cases]

# 3. Mide métricas
accuracy = sum(1 for r, tc in zip(results, test_cases)
               if evaluate(r, tc["expected_output"], tc["criteria"]))
               / len(test_cases)

10.1 System Prompt Design Patterns

Patrones probados para system prompts robustos

📝 Convención del curso: los prompts de ejemplo se muestran en castellano cuando estás en modo ES. En producción, escribir prompts en inglés suele dar 5-15% mejor calidad en los modelos abiertos (Llama, Qwen, Gemma, Mistral) — para frontier (Claude, GPT-5, Gemini, DeepSeek) la diferencia es menor pero existe. Adapta el idioma al modelo target.

1. Patrón: Guardrails First (límites primero)

# Establece límites claros al inicio
Eres un asistente médico. REGLAS CRÍTICAS:
- NUNCA diagnostiques — solo sugiere posibilidades.
- SIEMPRE recomienda consultar a un médico.
- Si te piden consejo de emergencia, responde con el 112 / número de emergencias.

2. Patrón: Persona + Expertise

Eres un desarrollador Python experto con 15 años de experiencia
en sistemas distribuidos. Valoras: corrección > ingenio,
legibilidad > concisión, código testeado > suposiciones.

3. Patrón: Output Template (plantilla de salida)

Responde siempre con este formato exacto:
<analysis>Tu razonamiento aquí</analysis>
<code>La implementación</code>
<tests>Casos de prueba que cubran edge cases</tests>

4. Patrón: Progressive Disclosure (revelación progresiva)

Paso 1: Haz preguntas aclaratorias si algo es ambiguo.
Paso 2: Explica tu enfoque antes de implementar.
Paso 3: Implementa la solución.
Paso 4: Verifica con tests y auto-revisión.

10.4 Fine-tuning vs Prompt Engineering

🎯 Prompt Engineering

  • ✅ Sin entrenamiento — cambios inmediatos
  • ✅ Itera en minutos/horas
  • ✅ Aprovecha el conocimiento general del modelo
  • ✅ Se adapta a nuevos casos sin reentrenar
  • ❌ Token cost recurrente (instrucciones largas)
  • ❌ Latencia extra con system prompts largos

🔧 Fine-tuning

  • ✅ Internaliza el comportamiento — no ocupa tokens
  • ✅ Más rápido en inferencia (sin prompt largo)
  • ✅ Consistente en dominios muy específicos
  • ❌ Necesita dataset curado (500+ ejemplos)
  • ❌ Coste de entrenamiento (tiempo + $)
  • ❌ Se vuelve obsoleto — requiere reentrenar
💡 Regla general (Anthropic): Empieza siempre con prompt engineering. Solo considera fine-tuning cuando hayas exprimido todas las técnicas de prompting y necesites consistencia extrema o ahorrar tokens.

🎯 Prompt Engineering

  • ✅ No training — instant changes
  • ✅ Iterate in minutes/hours
  • ✅ Leverages the model's general knowledge
  • ✅ Adapts to new cases without retraining
  • ❌ Recurring token cost (long instructions)
  • ❌ Extra latency with long system prompts

🔧 Fine-tuning

  • ✅ Internalizes behavior — no token cost
  • ✅ Faster inference (no long prompt)
  • ✅ Consistent in very specific domains
  • ❌ Needs a curated dataset (500+ examples)
  • ❌ Training cost (time + $)
  • ❌ Goes stale — requires retraining
💡 Rule of thumb (Anthropic): Always start with prompt engineering. Only consider fine-tuning when you've exhausted prompting techniques and need extreme consistency or token savings.
