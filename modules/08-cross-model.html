8.1 ¿Por qué agnóstico? 8.1 Why agnostic?

El reflejo natural cuando un proveedor te trata bien es casarte con él: API key única, framework familiar, equipo entrenado, dashboard que conoces. Y entonces, una mañana de 2024, alguien te dice "ah, OpenAI ha duplicado el precio de GPT-4 Turbo" o "Claude tiene un outage de 6 horas y nuestro chatbot está caído", y descubres en directo el coste del lock-in.

🧭 Las 5 razones del enfoque agnóstico
  1. Resiliencia operativa. Cualquier proveedor frontier ha tenido outages multi-hora en 2024-2026. Si tu producto depende de uno, tu producto se cae con él. Multi-proveedor con fallback es seguro de continuidad.
  2. Apalancamiento comercial. Cuando puedes mover el 80% de tu volumen a otro proveedor en 48h, el descuento que negocias cambia drásticamente. La amenaza creíble de migración es la herramienta de negociación más fuerte.

8.5 Fallback chains — resiliencia operativa 8.5 Fallback chains — operational resilience

Un fallback chain es una lista ordenada de modelos que se prueban secuencialmente: si el primero falla (timeout, rate-limit, 5xx, contenido rechazado), se intenta el siguiente. Es la pieza más importante de un sistema serio que dependa de LLMs.

Anatomía de un fallback chain bien diseñado

# config/fallback.yaml — patrón production
primary: claude-opus-4-7        # mejor calidad, latencia ~5-15s
secondary: gpt-5-5              # similar capacidad, otro proveedor (resiliencia geográfica)
tertiary: deepseek-v4-pro       # 30× más barato, calidad aceptable
emergency: qwen3-32b-local      # sin internet, sin coste, 80% calidad
disable_on_repeat_failures: 3   # si los primeros 3 retries fallan, salta al siguiente

retry:
  timeout_s: 30
  max_retries: 2
  backoff: exponential  # 1s, 2s, 4s
  retry_on:             # códigos que disparan fallback
    - timeout
    - 429        # rate limit
    - 5xx        # server errors
    - content_filter  # respuesta rechazada por safety

Patrones comunes de fallback

PatrónDescripciónCuándo usarlo
Same-tier fallbackClaude Opus → GPT-5.5 (mismo nivel calidad/coste)Resiliencia frente a outages, sin bajada de calidad
Down-tier fallbackClaude Opus → Claude Sonnet → DeepSeekSi el primario está caído, aceptar calidad menor para no caer del todo
Local emergencyCloud → Ollama localContinuidad cuando hay outage masivo (toda la red, todos los proveedores)
Quality fallbackSi la respuesta del primario falla evals → reintentar en secundarioQuality assurance — el modelo más barato suele bastar pero a veces falla
Cost-cap fallbackSi el coste mensual rebasa $X → bajar a tier inferior automáticamenteProducto con presupuesto fijo
⚠️ Errores comunes en fallback chains
  1. No probar el path de fallback en CI. El primer outage real va a ser la primera vez que ejercitáis ese código. Test con chaos monkey.
  2. Fallback infinito. Si tu chain tiene N modelos y todos fallan, eventualmente el caller verá un error claro — no quedarse colgado intentando.
  3. Diferente formato de respuesta sin normalizar. Si el primario devuelve JSON estricto y el fallback devuelve markdown, el código downstream se rompe. Normaliza siempre antes de devolver.
  4. Diferente coste sin alertar. Si el fallback es 50× más caro y nadie lo monitoriza, tu factura se dispara silenciosamente.
  5. Determinismo perdido. Misma query → distinto modelo → distinta respuesta. Si tu producto promete reproducibilidad, declara en logs qué modelo respondió cada request.

A fallback chain is an ordered list of models tried sequentially: if the first fails (timeout, rate-limit, 5xx, rejected content), the next is attempted. It's the single most important piece of a serious LLM-dependent system.

Anatomy of a well-designed fallback chain

# config/fallback.yaml — production pattern
primary: claude-opus-4-7        # best quality, ~5-15s latency
secondary: gpt-5-5              # similar capability, different provider (geo resilience)
tertiary: deepseek-v4-pro       # 30× cheaper, acceptable quality
emergency: qwen3-32b-local      # offline, free, 80% quality
disable_on_repeat_failures: 3   # if first 3 retries fail, jump to next

retry:
  timeout_s: 30
  max_retries: 2
  backoff: exponential  # 1s, 2s, 4s
  retry_on:             # codes that trigger fallback
    - timeout
    - 429        # rate limit
    - 5xx        # server errors
    - content_filter  # rejected by safety

Common fallback patterns

PatternDescriptionWhen to use
Same-tier fallbackClaude Opus → GPT-5.5 (same quality/cost tier)Resilience against outages, no quality loss
Down-tier fallbackClaude Opus → Claude Sonnet → DeepSeekIf primary is down, accept lower quality vs total downtime

⚠️ Common fallback chain mistakes
  1. Not testing the fallback path in CI. The first real outage will be the first time you exercise that code. Test with chaos monkey.
  2. Infinite fallback. If your chain has N models and all fail, the caller eventually needs a clean error — never hang trying.
  3. Different response format unnormalised. If primary returns strict JSON and fallback returns markdown, downstream breaks. Always normalise before returning.
  4. Different cost unalerted. If the fallback is 50× pricier and nobody monitors it, your bill silently explodes.
  5. Lost determinism. Same query → different model → different answer. If your product promises reproducibility, log which model answered each request.

8.7 Cross-model evaluation — comparar manzanas con manzanas 8.7 Cross-model evaluation — apples to apples

"Probé el mismo prompt en Claude y GPT, y Claude responde mejor" es la frase más común y la más errónea. Sin metodología, esa "intuición" mide más a tu sesgo de confirmación que a los modelos. La cross-model evaluation seria sigue 5 reglas.

📏 Las 5 reglas de la cross-model eval
  1. Mismo dataset — Mínimo 50 inputs representativos de tu caso real con respuestas "ground truth" preparadas. No pruebas con 5 ejemplos cherry-picked.
  2. Mismas instrucciones — Si tu prompt está afinado para Claude (XML, "respond in flowing prose"), GPT y Gemini pueden rendir peor por convención. Prepara dos prompts: el "common-denominator" (works in all) y el optimizado por modelo, y reporta ambos.
  3. Mismos parámetros — temperature, top_p, max_tokens iguales o explicitados en el reporte.
  4. Misma métrica — exact-match, contains, BLEU, LLM-as-Judge — define ANTES de mirar resultados. Si tu métrica es "calidad" subjetiva, usa LLM-as-Judge con un evaluador distinto a los modelos comparados.
  5. N suficiente — diferencias inferiores a ~3% en una muestra de 50 son ruido. Para diferencias finas necesitas 200+ ejemplos.

Plantilla mínima de eval

# eval_cross_model.py — runnable, ~80 LOC
import asyncio
from litellm import acompletion

DATASET = [
    {"input": "...", "expected": "...", "criterion": "contains"},
    # ... 50+ ejemplos
]

MODELS = ["anthropic/claude-opus-4-7", "openai/gpt-5", "deepseek/deepseek-chat"]

async def run_one(model, sample):
    r = await acompletion(model=model,
                          messages=[{"role": "user", "content": sample["input"]}],
                          temperature=0.0, max_tokens=512, seed=42)
    out = r.choices[0].message.content
    if sample["criterion"] == "contains":
        ok = sample["expected"].lower() in out.lower()
    elif sample["criterion"] == "exact":
        ok = sample["expected"].strip() == out.strip()
    return ok, out

async def main():
    results = {m: 0 for m in MODELS}
    for sample in DATASET:
        for m in MODELS:
            ok, _ = await run_one(m, sample)
            results[m] += int(ok)
    for m, c in results.items():
        print(f"{m}: {c}/{len(DATASET)} = {100*c/len(DATASET):.1f}%")

asyncio.run(main())
⚠️ Trampa frecuente: usar el mismo modelo (ej. GPT-4) como evaluador del comparativo entre Claude/GPT/Gemini. GPT-4 favorece outputs estilísticamente cercanos al suyo. Si haces LLM-as-Judge, el evaluador debe ser un modelo DISTINTO de los comparados, idealmente uno más capaz (Claude Opus 4.7 evalúa Sonnet vs GPT-5).

"I tried the same prompt in Claude and GPT, and Claude does better" is the most common and most wrong sentence. Without methodology, that "intuition" measures your confirmation bias more than the models. Serious cross-model evaluation follows 5 rules.

📏 The 5 rules of cross-model eval
  1. Same dataset — Minimum 50 inputs representative of your real case with prepared "ground truth" answers. Not 5 cherry-picked examples.
  2. Same instructions — If your prompt is tuned for Claude (XML, "respond in flowing prose"), GPT and Gemini may score worse for convention reasons. Prepare two prompts: the "common-denominator" (works everywhere) and the per-model optimised one, and report both.
  3. Same parameters — temperature, top_p, max_tokens identical, or explicit in the report.
  4. Same metric — exact-match, contains, BLEU, LLM-as-Judge — defined BEFORE looking at results. If your metric is subjective "quality", use LLM-as-Judge with an evaluator different from the compared models.
  5. Sufficient N — <3% gaps on a 50-sample run is noise. Fine differences need 200+ examples.
