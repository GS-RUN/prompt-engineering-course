21.1 ¿Qué es Cuantizar un Modelo?

Comprimir modelos para que quepan en tu GPU (o en tu móvil)

📐 De Floats a Integers

Un modelo sin cuantizar usa FP32 (32 bits por parámetro). Un modelo de 70B parámetros ocupa 280 GB — necesitas 4+ GPUs A100. La cuantización reduce la precisión de los pesos para que ocupen menos.

Niveles de Precisión

FormatoBits por parámetroModelo 7BModelo 70BCalidad relativa
FP32 (full precision)3228 GB280 GB100% (referencia)
FP16 / BF161614 GB140 GB~99.9%
INT887 GB70 GB~99.5%
INT4 (4-bit)43.5 GB35 GB~97-99%
INT3 (3-bit)32.6 GB26 GB~93-96%
INT2 (2-bit)21.75 GB17.5 GB~80-90% ⚠️

El "Punto Dulce"

# Trade-off calidad vs tamaño:

FP32 → Solo para entrenamiento. Nadie infiere en FP32.
FP16 → Standard para inference en servidor (vLLM, TGI).
INT8 → Buen balance. Casi idéntica calidad. 2x más pequeño.
INT4 → EL sweet spot. 97-99% calidad por 4x menos memoria.
       La mayoría de modelos "locales" usan 4-bit (GPTQ, AWQ, GGUF).
INT3 → Aceptable para modelos grandes (70B+) donde cada GB cuenta.
INT2 → Emergencia. Solo si no hay otra opción. Degradación notable.
💡 Dato clave: Un modelo de 70B en INT4 cabe en una sola RTX 4090 (24 GB). En FP16 necesitarías 2-3 GPUs. La cuantización es lo que democratiza los LLMs locales.
📐 From Floats to Integers

An unquantized model uses FP32 (32 bits per parameter). A 70B-parameter model takes 280 GB — you need 4+ A100 GPUs. Quantization reduces weight precision so they take less space.

Formato GGUF — El Estándar Local

# GGUF es el formato que usan Ollama y llama.cpp
# Nomenclatura típica:

q2_K    — 2-bit, calidad baja, ~90% del original
q3_K_M  — 3-bit medium, ~94%
q4_K_M  — 4-bit medium, ~97% ← RECOMENDADO para la mayoría
q5_K_M  — 5-bit medium, ~98.5%
q6_K    — 6-bit, ~99%
q8_0    — 8-bit, ~99.5%

# K-quants: cuantización MIXTA
# - Las capas de atención usan más bits (son más sensibles)
# - Las capas feed-forward usan menos bits
# K_M = balance calidad/tamaño. K_S = más pequeño. K_L = más calidad.

# Ejemplo con Ollama:

13.1 Cómo se Entrena un LLM

De texto crudo de internet a asistente conversacional — las 3 fases del entrenamiento

🏋️ Fase 1: Pre-training (Self-Supervised)
# Objetivo: Predecir el siguiente token
# Datos: Trillones de tokens de internet, libros, código
# Tiempo: Semanas/meses en miles de GPUs
# Coste: $10M - $500M+

Input:  "The capital of France is"
Target: "Paris"

# El modelo aprende: gramática, hechos, razonamiento, patrones
# NO aprende: a seguir instrucciones, a ser útil, a decir "no sé"
🔬 Fase 2: Post-Training (Alignment)
# Objetivo: Enseñar al modelo a ser útil, inofensivo y honesto
# Métodos: SFT → RLHF → DPO → Constitutional AI

1. SFT (Supervised Fine-Tuning): Entrenar con ejemplos de diálogo ideal
2. RLHF: Humanos rankean respuestas → Reward Model → Refuerzo
3. DPO: Optimización directa sin reward model separado
4. Constitutional AI: El modelo se auto-mejora siguiendo principios
🎓 Fase 3: Specialization (Opcional)
# Fine-tuning para dominios o tareas específicas
# Datos: 500 - 10K ejemplos de alta calidad
# Tiempo: Minutos/horas en 1-8 GPUs
# Coste: $10 - $1000 (LoRA/QLoRA)

Ejemplo: Fine-tunear Llama 4 para hablar solo en haikus,
o para extraer campos específicos de facturas en PDF.
💡 Dato clave: El pre-training es lo que hace al modelo "inteligente". El post-training es lo que lo hace "útil". Un modelo solo pre-entrenado (base model) completa texto pero no sigue instrucciones — necesita post-training para ser un asistente.
🏋️ Phase 1: Pre-training (Self-Supervised)
# Goal: Predict the next token
# Data: Trillions of tokens from internet, books, code
# Time: Weeks/months on thousands of GPUs
# Cost: $10M - $500M+

Input:  "The capital of France is"
Target: "Paris"

# The model learns: grammar, facts, reasoning, patterns
# Does NOT learn: to follow instructions, to be helpful, to say "I don't know"
🔬 Phase 2: Post-Training (Alignment)
# Goal: Teach the model to be helpful, harmless and honest
# Methods: SFT → RLHF → DPO → Constitutional AI

1. SFT (Supervised Fine-Tuning): Train on examples of ideal dialogue
2. RLHF: Humans rank responses → Reward Model → Reinforcement
3. DPO: Direct optimization without a separate reward model
4. Constitutional AI: The model self-improves following principles
🎓 Phase 3: Specialization (Optional)
# Fine-tuning for specific domains or tasks
# Data: 500 - 10K high-quality examples
# Time: Minutes/hours on 1-8 GPUs
# Cost: $10 - $1000 (LoRA/QLoRA)

Example: Fine-tune Llama 4 to speak only in haikus,
or to extract specific fields from PDF invoices.
💡 Key insight: Pre-training is what makes the model "smart." Post-training is what makes it "useful." A purely pre-trained model (base model) completes text but doesn't follow instructions — it needs post-training to become an assistant.

¿Cuándo Fine-tunear?

  • ✅ Tienes 500+ ejemplos de alta calidad de tu tarea específica
  • ✅ El prompting no llega: Has probado todas las técnicas y no alcanzas la calidad necesaria
  • ✅ Necesitas un estilo/tomo muy específico y consistente
  • ✅ Quieres reducir coste de tokens: El modelo aprende el comportamiento sin necesidad de prompts largos
  • ✅ Domínio muy especializado: Terminología médica, legal, o técnica que el modelo base no domina
  • ❌ Menos de 200 ejemplos: Overfitting casi garantizado
  • ❌ La tarea cambia frecuentemente: Re-entrenar es costoso
  • ❌ Quieres conocimiento nuevo: El fine-tuning NO añade datos factuales nuevos, solo cambia el comportamiento. Para añadir conocimiento, usa RAG.
⚠️ Fine-tuning NO es para añadir conocimiento nuevo. Es para cambiar COMPORTAMIENTO (estilo, formato, tono). Para datos factuales nuevos (ej: la documentación de tu API), usa RAG o prompt con contexto.

When to Fine-tune?

  • ✅ You have 500+ examples of high quality on your specific task
  • ✅ Prompting isn't enough: You've tried every technique and can't reach the needed quality
  • ✅ You need a very specific style/tone consistently
  • ✅ You want to reduce token cost: The model learns the behavior without long prompts
  • ✅ Highly specialized domain: Medical, legal or technical terminology the base model doesn't master
  • ❌ Fewer than 200 examples: Overfitting almost guaranteed
  • ❌ The task changes frequently: Retraining is expensive
  • ❌ You want new knowledge: Fine-tuning does NOT add new factual data, only changes behavior. To add knowledge, use RAG.
⚠️ Fine-tuning is NOT for adding new knowledge. It's for changing BEHAVIOR (style, format, tone). For new factual data (like your API docs), use RAG or prompt with context.
